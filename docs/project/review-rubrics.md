# Review rubrics

Use these criteria when reviewing or generating project docs (e.g. during bootstrap or feature passes).

## General

- **Clarity** — Language is clear and unambiguous; a new reader can understand intent.
- **Consistency** — Terminology and structure are consistent across docs; no contradictory statements.
- **Completeness** — Docs cover the scope implied by the source (e.g. issue body, feature spec); gaps are called out.
- **Structure** — Sensible headings, lists, and cross-references; easy to navigate.

## Bootstrap / initial docs

- **Vision and goals** — Vision and goals align with the problem statement and personas.
- **Requirements** — Business requirements are traceable to goals and personas.
- **Roadmap** — Phases and ordering are coherent with scope and dependencies.
- **Architecture** — Overview is consistent with requirements and standards; no orphan concepts.

## Optional

- **API / data / UI / integration** — If present, standards docs are consistent with architecture-overview and business-requirements.

Apply a practical first-pass review against these when generating the initial doc set; do not block on human approval for the bootstrap milestone.
